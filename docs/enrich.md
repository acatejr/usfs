# Enriching the Inventory (`enrich`)

## What it does

When datasets are first harvested from the three Forest Service sources, their
descriptions are inconsistent — some are long and technical, some are short,
some are missing entirely, and the wording varies from source to source. This
makes the inventory hard to search and compare.

The `enrich` command improves each dataset's entry so they all follow the same
clear format. For every dataset it adds:

- **A plain-language summary** — one or two sentences describing what the
  dataset is.
- **Topic tags** — a short list of reusable labels (for example, `wildfire`,
  `hydrology`, `timber harvest`) that make related datasets easy to group.
- **A data type** — what kind of data it is (a map, a table, a document, etc.).
- **Quality flags** — automatic notes about what might be missing from the
  original record (see below).

The summaries and topic tags are written by an AI language model that reads the
original Forest Service metadata. The quality flags are calculated directly from
the data by simple rules, with no AI involved.

## How to run it

From the project root:

```bash
uv run usfs enrich
```

Before this works, the inventory must already be loaded into the database. The
normal order is:

```bash
uv run usfs harvest    # download the source data and build the inventory
uv run usfs initdb     # set up the database
uv run usfs load       # load the inventory into the database
uv run usfs enrich     # add summaries, topics, data types, and quality flags
```

### It is safe to run more than once

`enrich` only processes datasets that have **not** been enriched yet. Datasets
that already have a summary are skipped. This means:

- If the command is interrupted, you can simply run it again and it picks up
  where it left off.
- If a dataset can't be processed on one run, it is left untouched and retried
  automatically next time — it is never left with a blank or broken summary.

## Reading the results

While it runs, the command reports its progress and a final count:

```
Enriching 412 rows ...
   25/412
   50/412
   ...
   Enriched 408/412 rows.
```

The two numbers in the final line matter:

| Number | Meaning |
|--------|---------|
| First (`408`) | How many datasets were successfully enriched this run. |
| Second (`412`) | How many datasets were eligible (had no summary yet). |

If the two numbers differ, the remaining datasets could not be processed this
time and will be retried the next time you run `enrich`. If everything is
already done, the command simply prints:

```
Nothing to enrich; all rows have summaries.
```

## What gets added to each dataset

| Field | Source | Description |
|-------|--------|-------------|
| **Summary** | AI | One or two sentences in plain language. |
| **Topics** | AI | 3–7 short, lowercase, reusable tags. |
| **Data type** | AI | One of: geospatial vector, raster, map service, tabular, document, or other. |
| **Quality flags** | Rules (no AI) | Notes on what the original record was missing. |

### Quality flags explained

Quality flags are automatic, objective notes about gaps in the *original*
Forest Service record — not judgments about the dataset's value:

- **`missing_narrative`** — the original record had no description, abstract, or
  statement of purpose.
- **`missing_keywords`** — the original record had no keywords.
- **`short_title`** — the title is very short (under 10 characters), which often
  signals an incomplete entry.

These flags help reviewers find records that may need attention from the data's
original publisher.

## A note on the AI step

The summaries and topics are generated by an AI language model. The model is
instructed to base its answer only on the metadata it is given and to respond in
a strict, predictable format. If the model ever returns something unusable, that
dataset is skipped and retried later rather than saved with bad content — so the
inventory never fills with unreliable text. The factual quality flags do not
depend on the AI at all.

## What to do with the results

After `enrich` runs, every dataset has a consistent, searchable description.
This is the step that makes the later search and question-answering features
useful. The next stage in the pipeline is usually:

```bash
uv run usfs embed      # index the enriched datasets so they can be searched
```

**The summary is what gets searched.** When `embed` indexes a dataset, it builds
its searchable text from the enriched summary first, then adds the topic tags,
title, and keywords. If a dataset has a summary, that summary — not the original
raw description — is the main text used. (If enrichment was skipped for a
dataset, `embed` falls back to its original narrative fields.) In short,
enriching directly improves the quality of later search and question-answering.

## Quick reference

| Question | Answer |
|----------|--------|
| What does it add? | A summary, topic tags, a data type, and quality flags. |
| What uses AI? | The summary, topics, and data type. Quality flags do not. |
| Is it safe to re-run? | Yes. It skips already-enriched datasets and retries failures. |
| What must run first? | `harvest` → `load`, then `enrich`. |
| What comes next? | `embed`, to make the inventory searchable. |

## Step-by-step example

Here is what happens, start to finish, when one dataset is enriched.

**1. A raw record exists in the inventory.** It was harvested and loaded, but its
description is sparse and inconsistent — and it has no summary yet:

```
Title:    Stream Crossings
Source:   fsgeodata
Abstract: Point locations where National Forest System roads cross streams.
Keywords: roads, streams, crossings
```

**2. You run the command:**

```bash
uv run usfs enrich
```

**3. The tool selects only un-enriched records.** Because this record has no
summary, it is included. Records that already have a summary are skipped.

**4. The original metadata is sent to the AI.** The tool gathers the title,
source, description, and keywords and asks the AI language model to return a
summary, topic tags, and a data type in a strict format.

**5. The AI responds, and the answer is checked.** A valid reply might be:

```
Summary:   Point locations where Forest Service roads cross streams, used for
           planning road maintenance and assessing aquatic impacts.
Topics:    roads, hydrology, stream crossings, infrastructure
Data type: geospatial vector
```

If the reply were unusable, the record would be left untouched and retried on a
later run — never saved with bad content.

**6. Quality flags are calculated separately.** Using simple rules (no AI), the
tool checks the *original* record for gaps. This one had a description, keywords,
and a normal-length title, so all flags are clear:

```
missing_narrative: false
missing_keywords:  false
short_title:       false
```

**7. The enriched record is saved.** The dataset now has a clear summary,
reusable topic tags, a data type, and quality flags — and is counted in the
final total:

```
   Enriched 1/1 rows.
```

**8. It is ready for the next step.** Now that the description is consistent and
searchable, the dataset can be indexed with `uv run usfs embed`.
